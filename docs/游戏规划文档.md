# Kitten Quest（小猫寻踪）游戏规划文档

**版本**：v2.0  
**日期**：2026-06-16  
**文档目标**：明确玩法规则、数学约束、关卡生成接口与实现边界，作为后续原型开发和程序实现基线。

---

## 1. 游戏定位

`Kitten Quest` 是一个 2D 网格逻辑解谜游戏。玩家需要在一个由彩色小方块组成的正方形场地中，找出所有隐藏的小猫。

核心体验不是拼手速，而是基于规则做排除、定位和确认：

- 每种颜色中恰好隐藏 1 只小猫
- 每只小猫所在的行、列都不会再出现其他小猫
- 小猫之间不能相邻，包含上下左右和四个对角
- 玩家可以先做标记，再进行最终确认
- 错误确认会扣生命值，形成有限试错压力

这套规则的手感接近“扫雷的标记反馈 + 数独/拉丁方类的排除推理”。

---

## 2. 规则澄清与统一定义

原始描述里有两个变量都被写成了 `n`：

- 一个表示场地边长
- 一个表示颜色数量，也等于小猫数量

这在实现时会造成歧义，因此文档统一为：

- `boardSize`：场地边长，场地大小为 `boardSize x boardSize`
- `catCount`：本关需要找到的小猫数量
- `colorCount`：颜色数量，固定等于 `catCount`

因此一关的核心输入建议是：

```ts
generateLevel(boardSize: number): LevelData
```

输入关卡场地大小 `boardSize` 后，颜色块种数和要找的小猫数量都固定等于 `boardSize`：

```ts
catCount = colorCount = boardSize
```

其中 `maxCats(boardSize)` 是当前边长下，在“不同行、不同列、且 8 邻域不相邻”的前提下，理论上最多能放多少只小猫。

---

## 3. 核心玩法规则

### 3.1 场地

- 场地是一个 `boardSize x boardSize` 的正方形网格
- 每个格子有一种颜色
- 场地中共有 `catCount` 种颜色
- 每种颜色至少出现 1 次
- 每种颜色的所有格子必须彼此相邻，形成一个连通色块
- 每种颜色中恰好有 1 个格子藏有小猫

说明：

- 颜色数量和小猫数量一一对应
- 某种颜色出现多少个格子，可以不固定相等
- 同色块之间允许任意形状扩展，但必须保持连通`r`n- 同色块生成时应优先向上下左右扩张，只在必要时才使用对角连接

### 3.2 小猫放置规则

假设某只小猫位于坐标 `(x, y)`，坐标原点在左上角，`x` 向右增长，`y` 向下增长。

所有小猫必须同时满足以下条件：

1. 每行最多 1 只小猫
2. 每列最多 1 只小猫
3. 每种颜色恰好 1 只小猫
4. 任意两只小猫不能处于彼此的 8 邻域内

也就是以下位置都不能再放其他小猫：

- `(x-1, y)`
- `(x+1, y)`
- `(x, y-1)`
- `(x, y+1)`
- `(x-1, y-1)`
- `(x+1, y-1)`
- `(x-1, y+1)`
- `(x+1, y+1)`

这正好对应你举的 `(3,5)` 示例。

### 3.3 玩家操作

单个格子的交互规则：

- 单击：切换标记状态
- 双击：确认该格子是小猫隐藏点

推荐标记状态循环：

`未标记 -> 怀疑是猫 -> 排除 -> 未标记`

双击结果：

- 如果选对：该格子翻出小猫，进度 `+1`
- 如果选错：扣 1 点生命值

### 3.4 生命值与胜负

- 每轮初始生命值默认为 `3`
- 每次双击确认错误，生命值 `-1`
- 生命值归零时，本轮失败
- 找出全部 `catCount` 只小猫时，本轮胜利

## 4. 数学分析：最大可放置小猫数量

### 4.1 约束本质

由于：

- 每行最多 1 只
- 每列最多 1 只
- 相邻 8 格不能再有猫

所以如果按行从上到下看，小猫所在列构成一个严格不重复的序列 `col[y]`，并且对相邻两行中的已选猫位满足：

```ts
abs(col[y] - col[y + 1]) >= 2
```

### 4.2 `maxCats(boardSize)` 的直观结论

在边长为 `N` 的棋盘上，满足上述约束时，最多可放置的小猫数量为：

```ts
maxCats(N) = N  // 当 N >= 4
```

可作为设计和程序实现中的上界。

### 4.3 结论说明

原因可以这样理解：

1. 小猫不能同行、同列
2. 相邻两行中的小猫列号差至少为 2
3. 若想放更多小猫，就必须尽量分散到不相邻的行上
4. 采用合法列置换并避开相邻行的相邻列，总是可以构造多只猫的合法解

构造示例：

- `N = 4`，最多 `4` 只，例如 `(0,0)`、`(1,2)`、`(2,0)`、`(3,2)`
- `N = 5`，最多 `5` 只
- `N = 6`，最多 `6` 只
- `N = 7`，最多 `7` 只

这个上界比原文“每行恰好 1 只、每列恰好 1 只”的设定更合理，因为你现在明确要求：

- 场地大小 `boardSize` 可变
- 小猫数量 `m` 固定等于场地边长 `boardSize`
- 颜色块种数也固定等于 `boardSize`

因此，正确的设计应当是“每行每列最多 1 只”，不是“每行每列恰好 1 只”。

## 5. 关卡生成设计

### 5.1 目标接口

建议的核心关卡生成函数：

```ts
type LevelData = {
  boardSize: number;
  catCount: number;
  lives: number;
  grid: TileData[][];
  cats: CatPlacement[];
  seed: string;
};

type TileData = {
  x: number;
  y: number;
  colorId: number;
};

type CatPlacement = {
  x: number;
  y: number;
  colorId: number;
};

function generateLevel(boardSize: number): LevelData
```

### 5.2 生成流程

推荐生成分为 4 步：

1. 计算本关最大可放置数量 `max = maxCats(boardSize)`
2. 令 `catCount = colorCount = boardSize`
3. 生成满足规则的小猫坐标集合
4. 根据小猫位置反向生成颜色分布网格

### 5.3 `catCount` 固定策略

`catCount` 不再随机，固定等于 `boardSize`；颜色块种数 `colorCount` 也等于 `boardSize`。

建议策略：

```ts
const max = maxCats(boardSize);
const min = Math.max(1, Math.floor(max * 0.6));
const catCount = randomInt(min, max);
```

参考结果：

- `boardSize = 6` -> `catCount = 6`
- `boardSize = 7` -> `catCount = 7`
- `boardSize = 8` -> `catCount = 8`

如果想更稳定地保证可玩性，建议加权到较大值：

```ts
// 更推荐
boardSize = 9 -> catCount = 9
```

这样每关的信息密度更高。

### 5.4 小猫位置生成算法

用回溯最稳妥。

目标是在 `boardSize x boardSize` 的网格中选出 `catCount` 个格子，使其满足：

- 坐标互不重复
- 行不重复
- 列不重复
- 两两不相邻（8 邻域）

伪代码：

```ts
function generateCatPlacements(boardSize: number, catCount: number): Point[] {
  const rows = shuffle(range(boardSize));
  const result: Point[] = [];
  const usedCols = new Set<number>();

  function backtrack(rowIndex: number): boolean {
    if (result.length === catCount) return true;
    if (rowIndex >= rows.length) return false;

    const row = rows[rowIndex];

    const remainingRows = rows.length - rowIndex;
    const need = catCount - result.length;
    if (remainingRows < need) return false;

    const cols = shuffle(range(boardSize));

    for (const col of cols) {
      if (usedCols.has(col)) continue;
      if (!isFarEnough(row, col, result)) continue;

      result.push({ x: col, y: row });
      usedCols.add(col);

      if (backtrack(rowIndex + 1)) return true;

      result.pop();
      usedCols.delete(col);
    }

    return backtrack(rowIndex + 1);
  }

  if (!backtrack(0)) {
    throw new Error("Failed to generate valid cat placements");
  }

  return result;
}
```

其中：

```ts
function isFarEnough(y: number, x: number, placed: Point[]): boolean {
  return placed.every((p) => {
    const dx = Math.abs(p.x - x);
    const dy = Math.abs(p.y - y);
    return !(dx <= 1 && dy <= 1);
  });
}
```

### 5.5 颜色网格生成算法

在得到所有小猫位置后，再生成颜色网格。

规则目标：

- 共 `catCount` 种颜色
- 每种颜色对应 1 只小猫
- 每种颜色的所有格子必须连通
- 猫所在格子必须属于它自己的颜色块

推荐做法：

1. 先为每只猫分配唯一颜色 `colorId`
2. 先把每只猫放入一个独立颜色块的核心格
3. 再从这些核心格向外扩张，直到所有格子都被填满

伪代码：

```ts
function buildColorGrid(
  boardSize: number,
  cats: CatPlacement[],
  catCount: number
): number[][] {
  const grid = Array.from({ length: boardSize }, () =>
    Array<number>(boardSize).fill(-1)
  );

  for (let i = 0; i < cats.length; i++) {
    const cat = cats[i];
    grid[cat.y][cat.x] = cat.colorId;
  }

  const frontier: Array<{ x: number; y: number; colorId: number }> = [];
  for (const cat of cats) {
    frontier.push({ x: cat.x, y: cat.y, colorId: cat.colorId });
  }

  while (true) {
    const emptyCells: Array<{ x: number; y: number }> = [];
    for (let y = 0; y < boardSize; y++) {
      for (let x = 0; x < boardSize; x++) {
        if (grid[y][x].colorId === -1) {
          emptyCells.push({ x, y });
        }
      }
    }

    if (emptyCells.length === 0) break;

    const next = frontier
      .flatMap((cell) => getNeighbors(cell.x, cell.y, boardSize).map((pos) => ({ ...pos, colorId: cell.colorId })))
      .filter((pos) => grid[pos.y][pos.x].colorId === -1);

    if (next.length === 0) {
      throw new Error("Failed to grow connected color blocks");
    }

    const choice = next[Math.floor(Math.random() * next.length)];
    grid[choice.y][choice.x] = { x: choice.x, y: choice.y, colorId: choice.colorId };
    frontier.push(choice);
  }

  return grid;
}
```

这种方式比“每色必须刚好出现 `boardSize` 次”更灵活，也更适配 `catCount < boardSize` 的关卡。

### 5.6 建议增加 seed

为了方便调试、复现和分享关卡，建议所有随机过程都挂在 `seed` 上：

```ts
generateLevel(boardSize: number, seed?: string): LevelData
```

如果未传入 `seed`，则函数内部自动生成。

## 6. 关卡合法性校验

关卡生成完成后，至少要做一次程序校验。

### 6.1 生成结果校验项

1. `catCount <= maxCats(boardSize)`
2. 小猫数量正确
3. 小猫颜色两两不同
4. 小猫不同行
5. 小猫不同列
6. 任意两只小猫不在 8 邻域内
7. 每种颜色至少出现 1 次
8. 每种颜色恰好对应 1 只小猫

### 6.2 可选：唯一解校验

如果你希望部分模式是“纯逻辑唯一解”，还可以增加求解器：

- 输入整张颜色网格
- 枚举每种颜色可能的猫位置
- 搜索所有满足规则的组合
- 统计解的数量

结果可分为三类：

- `0 解`：关卡非法，直接丢弃
- `1 解`：唯一解，可用于标准解谜模式
- `多解`：可用于休闲模式，允许试错

注意：按你的当前玩法，“有生命值”并不强制要求唯一解，因此唯一解不是 MVP 必需项。

## 7. 难度设计建议

### 7.1 难度来源

这个游戏的难度主要来自 4 个因素：

- `boardSize` 越大，观察成本越高
- `catCount` 越高，约束交叉越密集
- 颜色分布越均匀，越容易推理
- 颜色分布越离散，越容易出现干扰项
- 1-6 格的小颜色块越多、面积越小，玩家越容易定位对应小猫，整体难度越低

### 7.1.1 固定 100 关生成规则

项目需要内置 100 个固定关卡，使用离线脚本生成并提交到代码仓库。

固定关卡规则：

- 共 100 关
- 每关使用固定 `seed`
- 每关保存 `boardSize`、`catCount` 和每个颜色块的目标面积 `regionSizes`
- 运行时根据这些配方重新生成完整棋盘
- 每一关必须至少存在 1 个面积为 `1~6` 的颜色块
- 颜色块面积越小，小猫越容易被定位，难度越低
- 面积为 `1~6` 的颜色块越多，难度越低

推荐难度曲线：

- 第 1~25 关：`6x6`，教学和轻量推理
- 第 26~50 关：`7x7`，开始出现更多颜色块组合
- 第 51~75 关：`8x8`，标准中等难度
- 第 76~100 关：`9x9`，后期挑战
- 第 77~90 关：`8x8`，中等偏上
- 第 91~97 关：`9x9`，后期挑战


实现文件：

- `scripts/generate-campaign.mjs`：生成 100 关配方
- `src/game/campaign.generated.json`：固定关卡配方数据
- `scripts/check-campaign.ts`：校验 100 关是否全部满足规则
- `src/game/campaign.ts`：运行时根据配方生成关卡

校验要求：

- 100 关均能生成合法棋盘
- 每关所有颜色块都保持 8 邻域连通
- 每种颜色恰好 1 只小猫
- 任意两只小猫不同行、不同列、且不相邻
- 每关至少存在 1 个面积为 `1~6` 的颜色块

### 7.2 推荐难度区间

| 场地边长 | 最大猫数 | 推荐猫数 | 难度定位 |
| --- | --- | --- | --- |
| 4 | 2 | 2 | 教学 |
| 5 | 3 | 2~3 | 简单 |
| 6 | 3 | 3 | 简单到中等 |
| 7 | 4 | 3~4 | 中等 |
| 8 | 4 | 4 | 中等到困难 |
| 9 | 5 | 4~5 | 困难 |
| 10 | 5 | 5 | 困难 |

### 7.3 推荐模式拆分

1. 教学模式  
固定小盘面，固定种子，强调规则学习

2. 标准模式  
随机地图，3 点生命，可存在多解

3. 挑战模式  
更大盘面，有限提示，尽量筛唯一解

## 8. 交互与 UI 设计建议

### 8.1 HUD 信息

建议顶部固定显示：

- 当前关卡边长
- 当前需找猫数 / 已找到数量
- 剩余生命值
- 计时
- 重开按钮
- 返回按钮

### 8.2 棋盘格状态

每个 Tile 至少有 5 种状态：

1. 默认
2. 怀疑是猫
3. 已排除
4. 已确认是猫
5. 错误确认后的已翻开空格

### 8.3 操作反馈

- 单击标记：轻微缩放或描边变化
- 双击确认正确：出现猫咪动画
- 双击确认错误：扣血、红色闪烁、轻震动
- 通关：整盘高亮、猫咪庆祝动画、结算面板

### 8.4 移动端兼容

移动端不建议强依赖双击。

推荐两种方案：

1. 点按切换标记，长按确认
2. 选中格子后，在底部操作条中点“确认是猫”

第二种更稳，误操作率更低。

## 9. 技术实现建议

### 9.1 前端技术路线

本项目建议明确采用 `React + TypeScript + Vite + Zustand` 作为第一阶段实现基线。

各自职责：

- `React`：负责页面结构、组件划分和交互组织
- `TypeScript`：负责规则建模、关卡数据结构和生成器类型安全
- `Vite`：负责开发服务器、构建和基础工程化
- `Zustand`：负责游戏状态管理，例如当前关卡、生命值、标记状态和胜负状态
- `CSS Grid`：负责第一版棋盘布局

这套组合的优点：

- 起步快，适合先验证玩法
- 规则逻辑和 UI 分层清晰
- 后续如果要接入 Phaser，也可以保留当前的数据层和状态层

### 9.2 MVP 技术范围

MVP 阶段建议先不引入重型 2D 引擎，而是用 React 完成以下内容：

- 关卡生成
- 棋盘渲染
- 单击标记
- 双击确认
- 生命值计算
- 胜负判定
- 下一关 / 重开

这样可以先确认玩法是否成立，再决定是否有必要升级为更重的表现层方案。

### 9.3 后续可升级内容

当核心玩法确认成立后，再考虑加入：

- Phaser 3
- 更丰富的猫咪动画
- 粒子特效
- 音效系统
- 每日挑战
- 提示系统
- 统计与成就

### 9.4 后续引擎升级方向

如果后续希望加入更复杂的表现，例如：

- 小猫逐帧动画
- 粒子喷发
- 镜头震动
- 更细的输入反馈

可以在第二阶段引入 `Phaser 3` 作为渲染层，但不建议在第一版就绑定到 Phaser。

推荐方式是：

- 保留 `game/` 下的规则与生成逻辑
- 保留 `store/` 下的状态管理
- 仅替换 `GameBoard` 的渲染实现

## 11. 风险与优化建议

### 11.1 当前规则下的主要风险

1. 小盘面可能较快被完成
2. 颜色如果随机填得过散，玩家可能更多依赖试错而不是推理
3. 双击在移动端体验一般
4. 如果没有种子机制，调试和复盘会很麻烦

### 11.2 对应优化

1. 固定使用 `catCount = colorCount = boardSize` 保证信息密度
2. 加一个颜色分布优化步骤，让同色格子形成适度聚类
3. 移动端使用长按或底部确认按钮
4. 全部随机流程统一挂 seed

## 12. 结论

这套玩法是成立的，但原文档里最需要修正的是两点：

1. 不能再把“场地边长”和“小猫数量/颜色数量”都写成 `n`
2. 行列规则应该是“最多 1 只”，不是“每行每列恰好 1 只”

按你现在的新需求，正确的生成逻辑应该是：

- 输入 `boardSize`
- 先算 `maxCats(boardSize)`
- 固定使用 `catCount = colorCount = boardSize`
- 生成 `catCount` 只互不同行、互不同列、互不相邻的小猫
- 再根据这些小猫反向生成颜色棋盘

这样文档和算法才一致，后续实现也不会出现接口冲突。
